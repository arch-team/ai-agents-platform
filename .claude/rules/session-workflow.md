# 会话工作流规则 (Session Workflow)

> Claude Code 每次会话自动执行的三步循环协议

---

## §0 速查卡片

```
开始 → 读取 progress.md → 识别任务/依赖/遗留/变更积压 → 向用户汇报
执行 → 加载参考规范 → 检查前置依赖 → TDD 实现 → 质量检查 → 完成度审查(§2.3)
结束 → 更新 progress.md 六个区域
```

### progress.md 更新清单

| 区域 | 更新内容 | 何时更新 |
|------|---------|---------|
| 当前状态 | "下一步"指向下一个待开始任务 | 每次会话结束 |
| 模块状态 | 模块整体完成时更新状态 | 模块完成时 |
| 任务拆解表 | 已完成任务状态改为"已完成"，填写会话日期 | 任务完成时 |
| 变更积压表 | 已完成变更状态改为"已完成"，填写会话日期；更新统计 | 变更完成时 |
| 遗留事项 | 新增未完成项 / 清除已解决项 | 有变更时 |
| 近期会话 | 在表头插入新行（保留最近 5 条，超出删除最旧） | 每次会话结束 |

## §1 会话开始协议

### 必须执行

1. **读取** `docs/progress.md`（已完成 Milestone 的 `<details>` 折叠块**不展开**，除非用户明确要求回顾）
2. **识别**以下信息：
   - 当前 Milestone 和阶段
   - 待做任务列表（状态为"待开始"或"进行中"）
   - 遗留事项（上次会话未完成的问题）
   - 前置依赖关系
   - **变更积压状态**（各级别未完成数量，是否有 S0 阻断项）
3. **向用户汇报**当前状态摘要（含变更积压概况）

### 任务选择规则（优先级仲裁）

优先级从高到低：

1. **用户指定** → 执行用户指定的任务编号或变更编号（`#N` 或 `C-SX-Y`）
2. **进行中任务** → 优先继续未完成的任务或变更
3. **S0 阻断变更** → 有未完成 S0 时，必须优先处理（阻断 Milestone 推进）
4. **遗留事项** → 提醒用户并建议优先处理
5. **最小编号 Milestone 任务** → 无以上阻塞时，按编号顺序推进

**穿插规则**: 连续 2-3 个 Milestone 任务完成后，提醒用户考虑处理 S1/S2 变更
**S3 触发**: S3 变更不自动触发，需用户主动发起；但当 Milestone 任务依赖某 S3 决策时需提醒

### 前置依赖检查

- 读取任务表"依赖"列
- 如前置任务未完成 → 提示用户，建议先完成前置任务
- 如前置任务被阻塞 → 报告阻塞原因，请用户决策

### §1.1 变更感知子协议

**S0 阻断检查**:
- 有未完成 S0 变更时，向用户发出阻断提醒：`⚠️ S0 阻断: X 项未完成，建议优先处理后再推进 M5`
- 列出未完成 S0 的编号和描述

**S1/S2 穿插提醒**:
- 查阅"近期会话"表，统计连续 Milestone 类型会话数
- 连续 3 个 Milestone 会话后（无变更处理），在状态汇报中提醒：`💡 穿插建议: 已连续 N 个 Milestone 会话，建议穿插处理 S1/S2 变更`
- 优先推荐修复成本低、影响范围小的变更

**S3 阻塞检查**:
- 当 Milestone 任务（如 knowledge 模块）依赖某 S3 决策（如 C-S3-1 数据库选型）时，提醒：`🔒 S3 阻塞: 当前任务依赖 C-S3-1 决策，建议先完成技术选型`

**变更依赖检查**:
- 读取变更积压表"依赖"列，对待执行的变更检查前置变更是否已完成
- 如前置变更未完成 → 提示用户，建议先完成前置变更

**模块亲和性提示**:
- 当开始某模块的 Milestone 任务时，扫描变更积压表"影响范围"列
- 如该模块有未完成变更，在状态汇报中提示：`📦 模块亲和: [模块名] 有 N 个未完成变更 (编号列表)，建议在开发该模块任务时一并处理`

## §2 任务执行协议

### 执行前

1. 读取任务表中"参考规范"列指定的 rules 文件
2. 确认前置依赖已满足
3. 确认任务范围和交付物

### 执行中

- 编码任务遵循 TDD 工作流：先写测试 → 再写实现 → 运行测试验证
- 将任务状态更新为"进行中"

#### §2.1 开发中发现子协议 (In-Dev Discovery)

开发过程中发现不属于当前任务的问题时，根据严重程度处理：

- **阻断当前任务**（不修会导致当前任务无法完成）：
  - 立即记入变更积压表，分配 `C-D-N` 编号（D = Discovery，N 为递增序号）
  - 标记优先级（参照 S0-S4 标准），填写来源为"开发发现 + 当前任务编号"
  - 可选择立即修复或暂停当前任务先处理该发现

- **不影响当前任务**：
  - 记入"遗留事项"区，附带建议优先级
  - 会话结束时提示用户：`💡 本次发现 N 个新问题，是否升级为正式变更？`
  - 用户确认后，升级为 `C-D-N` 编号纳入变更积压表

### 执行后 — 质量检查

编码任务完成后，自动运行以下检查：

```bash
ruff check backend/
mypy backend/
pytest backend/tests/
```

**Milestone 任务**：全部通过 → 任务标记为"已完成"

**变更项**：通用检查通过后，还需读取 `improvement-plan.md` 中该变更的"验证标准"逐条确认（如 C-S0-5 需确认"停用用户持有有效 Token 时请求返回 401"）

#### §2.2 回归检测

质量检查中如有测试失败，区分处理：

- **当前修改模块的测试失败** → 正常修复循环，修复后重新运行
- **其他模块的既有测试失败**（回归信号）→ 暂停修复，向用户报告：
  - 回归影响范围（哪些模块、哪些测试）
  - 可能原因分析
  - 建议选项：(a) 继续修复使所有测试通过 (b) 回滚本次变更 (c) 创建 ADR 记录权衡
  - 由用户决策后继续

有失败项且非回归 → 修复后重新运行，直到通过

#### §2.3 完成度审查（质量检查通过后）

质量检查（ruff/mypy/pytest）通过后，**标记任务完成前**必须执行：

1. **计划逐条对照**: 重新读取实施方案中该 Task 的 `- [ ]` 子项，确认每项交付物存在
2. **参考模块结构 diff**: 将新模块与最近完成的同类模块做文件结构对比（`events.py`、`__init__.py` 导出、集成测试等）
3. **向用户汇报**: 列出已完成项 + 与参考模块的已知差异/简化（如有意跳过需说明理由）

**不执行此步骤就标记完成视为流程违规。**

详见 `backend/.claude/rules/checklist.md` §Task 完成验收

### ADR 触发条件

以下情况需要创建架构决策记录 (`docs/adr/00X-*.md`)：

- 面临两个以上可行技术方案需要选择
- 实际开发与现有规范产生冲突
- 引入新的外部依赖或第三方库
- 偏离 rules 中既定模式的设计决策

## §3 会话结束协议

### 必须更新 `docs/progress.md` 的六个区域

1. **当前状态**
   - 更新"下一步"为下一个待开始任务的描述
   - 如 Milestone 完成，更新里程碑状态
   - 更新变更积压摘要行（如有变更状态变化）

2. **模块状态**
   - 模块内所有任务完成 → 状态改为"已完成"
   - 模块有任务进行中 → 状态改为"进行中"

3. **任务拆解表**
   - 已完成任务：状态改为"已完成"，会话列填写当前日期
   - 部分完成任务：状态改为"进行中"，细节记入遗留事项

4. **变更积压表**
   - 已完成变更：状态改为"已完成"，会话列填写当前日期
   - 部分完成变更：状态改为"进行中"，细节记入遗留事项
   - 更新变更统计表中的"当前进度"列

5. **遗留事项**
   - 新增：本次未完成的工作、发现的问题、跨会话待处理项
   - 清除：本次已解决的遗留项

6. **近期会话**（在表头插入新行，保留最近 5 条，超出时删除最旧记录）
   - #：递增序号（最新为最大值）
   - 日期：当前日期
   - 类型：Milestone / 变更 / 审查 / 工作流优化 / 架构决策
   - 完成项：本次完成的具体工作（精简为 1-2 句）
   - 关键决策：本次做出的技术决策（如有，精简为关键词）

## §4 Milestone 验收协议

当 Milestone 所有实现任务完成后，执行验收：

### 自动化检查

```bash
ruff check backend/
mypy --strict backend/
pytest --cov-fail-under=85 backend/tests/
```

### 规范对照

1. 对照 `backend/.claude/rules/checklist.md` 逐项检查
2. 对照 `docs/strategy/roadmap.md` 中对应 Milestone 的验收标准

### 验收结果处理

- 全部通过 → 更新 Milestone 状态为"已完成"，开始拆解下一个 Milestone
- 有未通过项 → 记录到遗留事项，创建修复任务

## §5 反馈循环

开发过程中各类产出的归档位置：

| 产出类型 | 归档位置 | 触发条件 |
|---------|---------|---------|
| 任务进度 | `docs/progress.md` 任务拆解表 | 每次会话结束 |
| 变更进度 | `docs/progress.md` 变更积压表 | 变更完成时 |
| 变更来源 | `docs/strategy/improvement-plan.md` | 执行变更时读取详细方案；注入变更时提取行动项 |
| 开发发现 | `docs/progress.md` 变更积压表 (`C-D-N`) | 开发中发现问题且用户确认升级时（见 §2.1） |
| 架构决策 | `docs/adr/00X-*.md` | 见 §2 ADR 触发条件 |
| 规范修正 | `backend/.claude/rules/*.md` | 发现规范与实际不符 |
| 遗留事项 | `docs/progress.md` 遗留事项区 | 任务未完成或发现跨会话问题 |

# AI Agents Platform 愿景与使命

> 基于 Amazon Bedrock AgentCore 的企业内部 AI Agents 平台

---

## 1. 企业 AI Agent 的挑战

### 1.1 从实验到生产的鸿沟

当前企业在 AI Agent 领域面临一个普遍困境：**原型容易，生产难**。

根据行业数据，79% 的组织已开始试验 AI Agent，但近三分之二尚未实现企业级规模化部署。这一数据揭示了从实验到生产之间存在着巨大鸿沟。具体表现为：

**Agent 开发门槛高**
- 构建一个生产级 Agent 涉及模型选择、提示工程、工具集成、记忆管理、错误处理等多个维度
- 缺乏统一的开发范式，团队需要在 LangGraph、CrewAI、OpenAI SDK 等众多框架中做选择
- 每个业务场景需要从零构建，缺乏可复用的模板和最佳实践

**能力分散且重复建设**
- 不同业务团队各自构建 Agent，导致重复开发基础能力（身份认证、工具连接、知识库管理）
- 缺乏统一的工具集成层，同一个内部 API 需要被不同 Agent 项目分别对接
- Agent 之间无法通信和协作，形成一个个孤立的 AI 孤岛

**缺乏统一管理和可观测性**
- 75% 的技术领导者将治理列为 Agent 部署的首要挑战
- 各团队独立部署 Agent，缺乏统一的生命周期管理（创建、测试、部署、监控、下线）
- 无法回答基本问题：谁创建了什么 Agent？Agent 做了什么决策？出了问题如何追溯？
- 安全和合规无法统一管控，每个 Agent 各自处理权限和数据访问

**运维和成本不可控**
- Agent 运行时资源难以预估和优化
- 模型调用成本分散在不同团队，缺乏统一的成本归因和预算管理
- 缺乏自动化的质量评估，Agent 输出质量依赖人工抽检

### 1.2 企业内部场景的特殊需求

除了通用挑战外，企业内部 Agent 场景还面临独特需求：

| 需求维度 | 具体要求 | 通用方案的不足 |
|---------|---------|-------------|
| **数据安全** | 内部数据不出 VPC，敏感操作审计追踪 | 通用 SaaS 平台无法满足数据驻留要求 |
| **权限体系** | 对接企业 SSO/LDAP，按部门/角色控制 Agent 访问 | 开源框架缺乏企业权限管理 |
| **内部系统集成** | 对接 ERP、CRM、HR 等内部系统和私有 API | 通用平台缺少定制化集成能力 |
| **审批流程** | Agent 关键操作需要人工审批确认 | 大多数框架缺乏 Human-in-the-Loop 的业务集成 |
| **成本归因** | 按部门/项目精确归因模型调用成本 | 通用平台缺乏企业级成本管理 |
| **渐进采用** | 从单个团队试点到全企业推广的平滑路径 | 要么太简单（无法满足技术团队），要么太复杂（业务团队无法使用） |

## 2. 平台愿景

> **为企业构建统一的 AI Agent 基础设施，让智能 Agent 像微服务一样被创建、部署、管理和协作。**

### 2.1 愿景阐述

我们的愿景是打造一个**企业内部的 AI Agent 操作系统**。正如 Kubernetes 统一了容器编排，本平台将统一企业内部 AI Agent 的全生命周期管理 —— 从创建到部署，从监控到治理，从单 Agent 到多 Agent 协作。

这不是又一个通用 Agent 开发框架，而是面向企业内部场景的**垂直整合平台**：

- **对开发者**: 提供标准化的 Agent 开发范式和丰富的企业集成工具，让构建生产级 Agent 从数月缩短到数天
- **对业务团队**: 提供可视化界面和预构建模板，让业务分析师无需编码即可创建解决特定场景的 Agent
- **对平台管理员**: 提供统一的治理控制台，实现全企业 Agent 的可见、可控、可追溯

### 2.2 远期图景

```
第一阶段: Agent 基础设施（0 → 6 个月）✅ 已完成 (Phase 1-3)
├── 统一的 Agent 创建和部署平台
├── 基于 AgentCore 的安全运行环境
└── 基础的监控和管理能力

第二阶段: Agent 智能化平台（6 → 18 个月）✅ 已完成 (Phase 4)
├── 企业 Agent 模板市场
├── Agent 自动质量评估和优化
├── 跨 Agent 协作编排
└── 业务级可观测性仪表盘

第三阶段: Agent 驱动的企业智能（18 → 36 个月）🔄 进行中 (Phase 5)
├── 5A 自动化评估 Pipeline（Bedrock Model Evaluation）← M13 ✅
├── 5B 对话式 Agent Builder（Claude MCP + SSO/LDAP）← M14
├── 5C 多租户规模运营（AWS Cost Explorer + billing）← M15
└── Agent 生态系统（企业知识图谱 + 跨系统集成）
```

## 3. 平台使命

> **降低企业 AI Agent 从实验到生产的门槛，让每个业务团队都能快速创建和部署解决实际问题的智能 Agent。**

### 3.1 使命拆解

**"降低门槛"** — 不是降低技术含量，而是消除非必要的复杂性：
- 基础设施复杂性由 AgentCore 承担
- 框架选择复杂性由平台兼容层处理
- 安全合规复杂性由平台治理层统一管理
- 开发者只需关注业务逻辑和 Agent 行为

**"从实验到生产"** — 桥接原型和生产之间的鸿沟：
- 提供从 Playground 到生产的渐进路径
- 内置质量门控和安全审查
- 自动化的测试、评估和部署流水线

**"每个业务团队"** — 不仅仅是技术团队的工具：
- 开发者通过 API/SDK 构建复杂 Agent
- 业务分析师通过模板和可视化构建标准 Agent
- 所有人通过统一控制台查看和管理 Agent

**"解决实际问题"** — 以结果为导向，而非技术导向：
- 衡量 Agent 的业务价值（效率提升、成本降低、质量改善）
- 提供业务模板（客服、数据分析、流程自动化等）
- 持续收集反馈，优化 Agent 表现

## 5. 核心价值主张

### 5.1 四大支柱

#### 支柱一: Bedrock AgentCore 深度集成

AgentCore 是 AWS 为 AI Agent 提供的最完整的企业级基础设施。我们不重建这些能力，而是在其上构建价值：

| AgentCore 能力 | 本平台增值 |
|---------------|----------|
| Runtime（MicroVM 隔离运行） | 一键部署 + 自动扩缩 + 运行状态监控 |
| Memory（短期+长期记忆） | 企业知识库集成 + 跨 Agent 知识共享 |
| Gateway（API → MCP 转换） | 内部 API 目录管理 + 自助注册工具 |
| Identity（OAuth/IAM） | 企业 SSO 集成 + 部门级权限隔离 |
| Observability（OTel 追踪） | 业务维度仪表盘 + 告警 + 成本归因 |

#### 支柱二: Claude Agent SDK 驱动的 Agent 开发

Claude Agent SDK 是对 Claude Code CLI 的编程封装，通过 `query()` API 让开发者以代码方式调用 Claude Code 的全部能力（工具、Agent 循环、上下文管理）：

- **内置工具执行**: Read/Write/Edit/Bash/Glob/Grep 等工具开箱即用，Agent 无需自行实现工具调用
- **Hooks 治理机制**: PreToolUse/PostToolUse 回调，实现企业级审计、权限拦截和行为监控
- **Subagents 协作**: 通过 Task 工具生成专业化子 Agent，支持复杂任务分解
- **MCP 原生集成**: 连接企业内部系统（数据库、API、工具），支持 stdio/HTTP/SSE 三种传输
- **Session 管理**: 会话恢复与分叉，支持长期任务和多轮交互
- **模型灵活性**: 通过 Bedrock 统一接口支持多模型，不锁定单一供应商

#### 支柱三: DDD 企业架构

采用 DDD（领域驱动设计）+ Modular Monolith + Clean Architecture 的企业级架构：

- **领域模型清晰**: Agent、Tool、Knowledge、Workflow 等核心领域明确边界
- **模块化演进**: 从单体开始，按需拆分微服务，避免过早优化
- **业务语言统一**: 领域通用语言贯穿代码、API、文档和 UI
- **可测试性**: 清晰的依赖关系和接口抽象，便于单元测试和集成测试

#### 支柱四: 企业治理与业务赋能

平台的核心价值不在于提供 Agent 运行时（AgentCore + Claude Agent SDK 已覆盖），而在于三层企业级增值：

- **企业治理**: 多租户资源隔离（按团队/项目）、平台级 RBAC 与审批工作流、成本归因与预算管理、合规审计报告
- **开发者增效**: Agent 配置管理与版本控制、基于 Claude Code Skills 的模板市场、企业 MCP 服务器目录与审批
- **业务赋能**: 企业数据深度集成（数据库、文档、IM）、内部知识库作为 Agent 知识来源、面向非技术用户的可视化 Agent 构建器

### 5.2 价值主张总结

> **"让企业内部的每个团队都拥有构建和管理 AI Agent 的能力，同时保证安全、合规和可控。"**

核心差异化：

| 对比维度 | 通用 Agent 框架 | 通用低代码平台 | 本平台 |
|---------|--------------|------------|-------|
| 目标用户 | 开发者 | 非技术人员 | 全层次（开发者+分析师+管理员） |
| 核心场景 | 任意 Agent 开发 | 简单应用搭建 | 企业内部业务 Agent |
| 安全模型 | 基础或无 | 平台级 | 企业级（SSO+RBAC+审计） |
| 运维管理 | 自行处理 | 平台托管 | 企业级治理 |
| 成本管理 | 无 | 平台级 | 部门级成本归因 |

## 6. 差异化分析

### 6.1 与云厂商原生服务的差异

**vs 直接使用 AWS Bedrock AgentCore**

AgentCore 提供的是基础设施能力，而非业务平台。直接使用 AgentCore 意味着：
- 需要自行构建 Agent 管理界面和 API
- 需要自行实现企业权限、审批流等业务逻辑
- 需要自行构建可视化监控和成本报告
- 每个团队需要独立学习和使用 AgentCore API

本平台在 AgentCore 之上提供了企业内部需要的完整业务层，使 AgentCore 的能力能够被非基础设施团队使用。

**vs Azure AI Foundry / Vertex AI Agent Builder**

这些是通用云服务，面向所有客户场景：
- 无法针对特定企业的内部流程深度定制
- 云厂商锁定（选择 Azure 就绑定 Microsoft 生态）
- 缺乏企业内部特有的治理需求（部门隔离、内部审批等）

### 6.2 与开源框架的差异

**vs LangGraph / CrewAI / OpenAI Agents SDK**

开源框架提供的是 Agent 编排能力，而非企业平台：
- 缺乏统一的部署和运维管理
- 缺乏企业级安全和权限体系
- 缺乏可观测性和成本管理
- 需要开发团队自行构建所有平台层能力

本平台**兼容而非替代**这些框架。开发者可以使用自己熟悉的框架开发 Agent，通过本平台统一部署和管理。

### 6.3 与低代码平台的差异

**vs Dify.ai / Coze / Flowise**

低代码平台面向通用市场，追求广泛适用性：
- 企业级治理能力不足（权限粒度、审计深度、合规报告）
- 无法深度定制集成企业内部系统
- 复杂 Agent 编排能力受限
- 数据驻留和安全隔离能力不足

本平台聚焦企业内部场景，在治理、集成和安全方面远超通用低代码平台。

## 7. 为什么自建

### 7.1 自建 vs 直接使用方案的对比

| 方案 | 优势 | 劣势 | 适用条件 |
|------|------|------|---------|
| **直接使用 SaaS（Dify/Coze）** | 零开发成本，立即可用 | 数据出境风险，定制能力弱，治理不足 | 非敏感场景的快速验证 |
| **直接使用 AgentCore** | 企业级基础设施，AWS 托管 | 需自建业务层，使用门槛高 | 纯技术团队的特定项目 |
| **使用开源框架 + 自运维** | 最大灵活性，无锁定 | 全栈自研，运维成本高 | 有强大技术团队的场景 |
| **自建平台（本方案）** | 企业适配，统一治理，渐进演进 | 前期投入较高，需持续迭代 | 有多团队 Agent 需求的企业 |

### 7.2 自建的核心理由

**1. 数据安全的刚性需求**

企业内部数据（业务数据、客户信息、财务数据等）不能流向外部 SaaS 平台。自建平台运行在企业自有的 AWS VPC 内，数据全程可控：
- 所有 Agent 运行在企业 VPC 内
- 通过 PrivateLink 访问 Bedrock 模型，数据不经过公网
- 操作日志和审计数据完全自有

**2. 企业定制化的必然选择**

每个企业的内部系统、组织架构、审批流程都不同。通用平台无法提供：
- 与企业 SSO/LDAP 的深度集成
- 按组织架构的 Agent 权限隔离
- 符合企业特定合规要求的审计机制
- 对接企业内部系统的专属工具集

**3. 成本效益的长期考量**

短期看自建投入更高，但长期来看：
- 多个团队共享平台能力，避免重复建设
- 统一的基础设施降低边际成本
- 自主掌握演进节奏，不受外部定价策略影响
- 积累的企业内部 Agent 模板和最佳实践是持续资产

**4. 战略自主权**

- 不依赖单一外部平台的功能迭代
- 可以根据企业实际需求调整功能优先级
- 核心能力内化，形成组织的 AI Agent 能力

### 7.3 自建的风险和挑战

| 风险 | 说明 | 缓解策略 |
|------|------|---------|
| **前期投入较高** | 需要投入开发资源构建平台 | MVP 先行，聚焦核心能力；依托 AgentCore 减少基础设施工作 |
| **技术复杂度** | AI Agent 技术栈演进快，需持续跟进 | 架构上保持框架无关，降低替换成本 |
| **团队能力要求** | 需要同时掌握 AI/ML 和企业架构能力 | 明确分工，AI 部分利用成熟框架，平台部分聚焦工程化 |
| **推广挑战** | 需要说服内部团队采用新平台 | 从"内部冠军团队"试点开始，用成功案例驱动推广 |
| **维护成本** | 持续迭代和运维的长期投入 | 精简架构，自动化运维，控制复杂度增长 |

### 7.4 决策依据

**建议自建的条件**（本项目满足以下条件）:
- 企业有多个团队同时需要 AI Agent 能力
- 数据安全和合规有刚性要求
- 已有 AWS 技术栈和基础设施投入
- 有 Python/FastAPI 后端开发能力
- 期望通过 AI Agent 解决多个业务场景

**不建议自建的条件**:
- 仅需单个 Agent 解决特定问题
- 没有持续投入的资源和意愿
- 对数据安全没有严格要求
- 没有 AWS 技术栈积累

## 8. 平台设计原则

### 8.1 核心设计原则

#### 原则一: 基础设施下沉，业务上浮

> 不重复建设已有的基础设施能力，将精力集中在业务适配层。

- AgentCore 负责运行时、记忆、身份、可观测性等基础设施
- 本平台负责企业管理、用户体验、业务模板等业务能力
- 清晰的职责边界避免与 AWS 服务功能重叠

#### 原则二: 框架兼容，不做绑定

> 支持主流 Agent 框架，让开发者用熟悉的工具，平台提供统一的部署和管理层。

- 兼容 LangGraph、CrewAI、OpenAI Agents SDK 等主流框架
- 通过 AgentCore Runtime 统一部署，不关心 Agent 用什么框架开发
- Claude Agent SDK 作为推荐但非强制的开发选择

#### 原则三: 分层用户体验

> 不同角色需要不同的交互方式，不要用一个界面试图满足所有人。

- 开发者: API/SDK + CLI，编程式体验
- 业务分析师: 可视化 Builder + 模板，低代码/无代码体验
- 管理员: 控制台 + 仪表盘，管理式体验

#### 原则四: 渐进式采用

> 从简单开始，按需增加复杂度。不要在第一天就要求用户学习全部平台能力。

- Day 1: 部署一个简单 Agent（分钟级）
- Week 1: 连接工具和知识库
- Month 1: 多 Agent 协作和业务集成
- Quarter 1: 全企业治理和规模化

#### 原则五: 可观测性优先

> 看不见的 Agent 不可控。从第一天就将可观测性作为一等公民。

- 每个 Agent 的每次推理、工具调用、决策过程都可追踪
- 业务维度的指标（成功率、响应时间、用户满意度）
- 成本维度的指标（按 Agent、按部门、按模型的使用量和费用）

#### 原则六: 安全即默认

> 安全不是功能，是默认行为。所有 Agent 在创建时就具备企业级安全能力。

- 默认在 VPC 内运行
- 默认启用身份认证和权限检查
- 默认记录操作审计日志
- 默认加密敏感数据

### 8.2 技术原则

| 原则 | 说明 | 实践 |
|------|------|------|
| **API First** | 所有功能先有 API，再有 UI | REST API → SDK → CLI → Web UI |
| **Event Driven** | 核心流程基于事件驱动 | Agent 生命周期事件 → 通知/审计/集成 |
| **Configuration over Code** | 尽可能通过配置而非代码定义 Agent 行为 | YAML/JSON Agent 定义 → 运行时解释执行 |
| **Immutable Deployment** | Agent 版本不可变，通过版本管理变更 | 版本号 → 灰度发布 → 回滚能力 |
| **Fail Safe** | Agent 失败时安全降级而非级联故障 | 超时保护 + 重试策略 + 人工兜底 |

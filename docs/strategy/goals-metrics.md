# 目标与指标体系 (Goals & Metrics)

> AI Agents Platform - 基于 Amazon Bedrock AgentCore 的企业内部 AI Agents 平台

---

## 1. 指标设计理念

### 1.1 为什么需要结构化的指标体系

AI Agents Platform 作为企业内部平台，面临与商业 SaaS 产品截然不同的挑战：没有直接的营收指标作为反馈信号，用户是内部员工而非付费客户，成功与否取决于是否真正提升了组织效率。因此，需要一套结构化的指标体系来回答三个核心问题：

1. **平台是否在被使用？** -- 采纳率和活跃度反映平台的实际价值
2. **平台运行是否健康？** -- 技术指标确保可靠的用户体验
3. **平台是否值得持续投入？** -- 成本效率指标支撑资源决策

### 1.2 企业内部平台指标 vs 商业 SaaS 指标

| 维度 | 商业 SaaS | 企业内部平台 |
|------|-----------|-------------|
| **核心驱动力** | 营收增长 (ARR/MRR) | 组织效率提升 |
| **用户获取** | CAC、转化漏斗 | 内部推广、团队采纳率 |
| **留存衡量** | 付费留存、流失率 | 持续使用率、场景覆盖 |
| **成功标准** | PMF、NRR | 效率提升比、开发时间节省 |
| **成本关注** | 毛利率、单位经济模型 | 基础设施成本、LLM 调用成本 |
| **反馈信号** | 付费意愿 | 主动使用意愿 |

### 1.3 指标设计原则

本指标体系遵循 SMART 原则，并针对企业内部平台做了适配：

- **Specific（具体）**：每个指标有明确的计算公式和数据来源
- **Measurable（可量化）**：所有指标均可通过系统数据自动采集
- **Achievable（可达成）**：分阶段设定目标，基于行业基准和团队规模
- **Relevant（相关）**：指标直接关联平台价值和组织目标
- **Time-bound（有时限）**：每个阶段有明确的时间窗口和检查点

## 2. 北极星指标 (North Star Metric)

### 2.1 推荐指标：周活跃 Agent 数量 (Weekly Active Agents, WAA)

**定义**：在过去 7 天内至少成功执行 1 次任务的 Agent 数量。

```
WAA = COUNT(DISTINCT agent_id)
      WHERE execution_status = 'success'
      AND execution_time >= NOW() - INTERVAL 7 DAY
```

### 2.2 为什么选择这个指标

选择 WAA 作为北极星指标，基于以下考量：

1. **反映真实价值交付**：Agent 被创建但未使用说明价值未实现，WAA 只统计"活跃"Agent，确保衡量的是实际产出
2. **关联用户和平台双向价值**：WAA 高意味着更多团队在持续使用平台解决实际问题
3. **可分解且可行动**：WAA 可以分解为创建量、激活率、持续使用率等子指标，每个子指标都可以对应具体的改进动作
4. **适合企业内部场景**：不依赖营收数据，直接反映平台在组织内的渗透深度

### 2.3 北极星指标的分解逻辑

```
WAA (周活跃 Agent) = 新创建 Agent 数 x 激活率 + 存量 Agent 数 x 周留存率

其中：
- 激活率 = 创建后 7 天内执行过至少 1 次任务的 Agent / 新创建 Agent 总数
- 周留存率 = 本周活跃且上周也活跃的 Agent / 上周活跃 Agent 总数
```

**分解树**：

```
WAA (周活跃 Agent)
├── 供给侧：Agent 创建
│   ├── 创建者数量（有多少人在创建 Agent）
│   ├── 人均创建数（创建效率）
│   └── 模板使用率（降低创建门槛）
├── 需求侧：Agent 使用
│   ├── 任务触发频率（Agent 被调用的频次）
│   ├── 任务成功率（Agent 能否完成任务）
│   └── 场景覆盖率（Agent 覆盖的业务场景）
└── 留存侧：持续活跃
    ├── Agent 周留存率
    ├── 用户回访率
    └── Agent 迭代频率（Agent 被优化的次数）
```

### 2.4 如何衡量和追踪

| 追踪维度 | 数据来源 | 采集频率 |
|---------|---------|---------|
| Agent 执行记录 | `agent_executions` 表 | 实时 |
| Agent 创建记录 | `agents` 表 | 实时 |
| 用户活动记录 | API 访问日志 | 实时 |
| WAA 聚合值 | 定时聚合任务 | 每日 |

**告警阈值**：WAA 周环比下降超过 20% 时触发告警，需在 48 小时内分析原因。

## 3. 分层指标体系

### 3.1 平台效能指标

衡量平台核心能力的输出和效率。

| 指标 | 计算公式 | 数据来源 | 采集频率 |
|------|---------|---------|---------|
| **Agent 总量** | `COUNT(agents)` | `agents` 表 | 实时 |
| **月新增 Agent 数** | `COUNT(agents WHERE created_at IN 当月)` | `agents` 表 | 日聚合 |
| **日任务执行量** | `COUNT(executions WHERE date = 当日)` | `agent_executions` 表 | 实时 |
| **Agent 场景覆盖率** | `已覆盖业务场景数 / 已识别可覆盖场景总数 x 100%` | 场景登记表（人工维护） | 月度 |
| **Agent 模板使用率** | `通过模板创建的 Agent 数 / Agent 总创建数 x 100%` | `agents` 表 (template_id 字段) | 周聚合 |
| **平均 Agent 构建时间** | `AVG(first_successful_execution_time - agent_created_time)` | `agents` + `agent_executions` 表 | 周聚合 |

**关键说明**：
- "Agent 场景覆盖率"中的"可覆盖场景"由产品团队与业务团队共同维护场景清单，按季度更新
- "平均 Agent 构建时间"衡量从创建到首次成功执行的端到端时间，反映平台易用性

### 3.2 技术健康指标

确保平台稳定可靠运行的技术底线。

| 指标 | 计算公式 | SLA 目标 | 告警阈值 |
|------|---------|---------|---------|
| **API 可用性** | `(1 - 故障时间/总时间) x 100%` | >= 99.9% (月) | < 99.5% |
| **P50 响应延迟** | 50th percentile of `response_time` | <= 200ms | > 500ms |
| **P95 响应延迟** | 95th percentile of `response_time` | <= 1s | > 2s |
| **P99 响应延迟** | 99th percentile of `response_time` | <= 3s | > 5s |
| **Agent 任务成功率** | `成功执行数 / 总执行数 x 100%` | >= 95% | < 90% |
| **系统错误率 (5xx)** | `5xx 响应数 / 总请求数 x 100%` | <= 0.1% | > 0.5% |
| **测试覆盖率** | 各层覆盖率加权平均 | >= 85% | < 80% |
| **部署频率** | `COUNT(deployments) / 工作日数` | >= 1次/周 | < 1次/2周 |
| **部署成功率** | `成功部署数 / 总部署数 x 100%` | >= 95% | < 90% |

**补充说明**：
- API 可用性排除计划维护窗口（需提前 24 小时通知）
- 响应延迟指标不含 LLM 推理时间，LLM 推理延迟单独追踪
- 测试覆盖率分层目标：Domain 95%、Application 90%、Infrastructure 80%、Presentation 80%

### 3.3 用户采纳指标

衡量平台在组织内的渗透和接受程度。

| 指标 | 计算公式 | 数据来源 |
|------|---------|---------|
| **内部团队采纳率** | `已使用平台的团队数 / 公司总团队数 x 100%` | 用户数据 + HR 团队数据 |
| **首次 Agent 创建时间 (TTFA)** | `用户首次成功创建 Agent 的时间 - 用户注册时间` | `users` + `agents` 表 |
| **用户满意度 (CSAT)** | 季度调查问卷评分 (1-5 分) | 问卷系统 |
| **月活跃用户数 (MAU)** | `COUNT(DISTINCT user_id) WHERE last_active >= NOW() - 30 DAY` | 用户活动日志 |
| **用户月留存率** | `本月活跃且上月也活跃的用户数 / 上月活跃用户数 x 100%` | 用户活动日志 |
| **文档使用率** | `文档页面月访问 UV / MAU x 100%` | 文档系统分析 |

**关键说明**：
- TTFA (Time to First Agent) 是衡量上手难度的关键指标，目标是让新用户在 30 分钟内完成首个 Agent 创建
- CSAT 调查不宜过于频繁，建议每季度一次，配合定性访谈
- "内部团队"的定义需与 HR 部门对齐，按部门或小组为最小单位

### 3.4 成本效率指标

确保平台投入产出比合理。

| 指标 | 计算公式 | 数据来源 |
|------|---------|---------|
| **单 Agent 月运行成本** | `月度总基础设施成本 / 月活跃 Agent 数` | AWS Cost Explorer + 平台数据 |
| **LLM Token 成本（按模型）** | `SUM(input_tokens x 单价 + output_tokens x 单价)` 按模型分组 | Bedrock 账单 + 调用日志 |
| **LLM Token 成本（按任务）** | `SUM(token_cost)` 按任务类型分组 | 调用日志 |
| **月度基础设施总成本** | AWS 月度账单中平台相关资源总和 | AWS Cost Explorer (CostCenter: ai-platform) |
| **人均基础设施成本** | `月度总成本 / MAU` | AWS Cost Explorer + 用户数据 |
| **成本效率比** | `估算节省的人工时间成本 / 平台总成本` | 效率评估 + AWS 账单 |

**关键说明**：
- 成本数据通过 AWS Cost Explorer 按 `CostCenter: ai-platform` 标签过滤
- LLM Token 成本需按 Claude (Sonnet/Haiku/Opus)、其他模型分别统计
- "估算节省的人工时间成本"通过抽样调查获取：统计 Agent 自动化前后完成同类任务的时间差，乘以人力成本

## 4. 分阶段量化目标

### 4.1 Phase 1 MVP (0-3 月)

**阶段重点**：验证核心流程，跑通端到端链路。

| 类别 | 指标 | 目标值 |
|------|------|-------|
| **功能** | 核心模块上线 | auth + agents + execution 3 个模块 |
| **功能** | 端到端场景验证 | >= 1 个场景完整跑通 |
| **功能** | Agent 模板数 | >= 2 个基础模板 |
| **技术** | API 可用性 | >= 99% (dev/staging 环境) |
| **技术** | P95 响应延迟 | <= 2s |

| **技术** | 测试覆盖率 | >= 85% |

## 5. 指标采集与看板

### 5.1 数据采集方案

| 数据类型 | 采集方式 | 存储位置 | 工具 |
|---------|---------|---------|------|
| **API 请求指标** | OpenTelemetry 中间件自动采集 | CloudWatch Metrics | structlog + OTEL |
| **Agent 执行数据** | 业务逻辑层记录 | Aurora MySQL | Application Service |
| **用户行为数据** | API 访问日志 + 前端埋点 | CloudWatch Logs | structlog |
| **基础设施成本** | AWS Cost Explorer API | S3 (成本报告) | Cost Explorer |
| **LLM 调用数据** | Bedrock 调用日志 | CloudWatch Logs | Bedrock Logging |
| **用户满意度** | 季度问卷调查 | 问卷平台 | 人工发起 |

### 5.2 监控看板设计

建议搭建三级看板，面向不同受众：

**Level 1 - 管理层看板（月度/季度）**
- 北极星指标 WAA 趋势图
- 团队采纳率热力图
- 成本效率比趋势
- 阶段目标完成度（进度条）

**Level 2 - 产品团队看板（每周）**
- Agent 创建/激活漏斗
- 模板使用排行
- TTFA 分布图
- 用户活跃度趋势（DAU/WAU/MAU）
- 场景覆盖进展

**Level 3 - 工程团队看板（实时/每日）**
- API 可用性和延迟（P50/P95/P99）
- 错误率和错误分类
- Agent 任务成功率
- LLM Token 消耗和成本
- 部署频率和成功率
- 基础设施资源利用率

**推荐工具**：CloudWatch Dashboard 作为主看板，配合 Grafana 做自定义可视化。

### 5.3 告警规则

| 告警级别 | 触发条件 | 响应时间 | 通知方式 |
|---------|---------|---------|---------|
| **P0 - 紧急** | API 可用性 < 99%、系统完全不可用 | 15 分钟 | 电话 + 即时消息 |
| **P1 - 严重** | Agent 任务成功率 < 85%、P95 延迟 > 5s | 1 小时 | 即时消息 |
| **P2 - 警告** | WAA 周环比下降 > 20%、错误率 > 0.5% | 24 小时 | 邮件 + 即时消息 |
| **P3 - 关注** | 成本超预算 10%、TTFA 超目标 50% | 下个工作日 | 邮件 |

## 6. 指标治理

### 6.1 指标评审机制

| 评审活动 | 频率 | 参与者 | 输出 |
|---------|------|-------|------|
| **周指标回顾** | 每周 | 工程团队 | 技术健康周报、异常分析 |
| **月度指标评审** | 每月 | 产品 + 工程 | 指标趋势分析、改进计划 |
| **季度指标复盘** | 每季度 | 管理层 + 产品 + 工程 | 阶段目标评估、指标体系调整 |
| **阶段里程碑评审** | Phase 切换时 | 全体利益相关者 | 成功标准验证、下阶段规划 |

### 6.2 指标调整流程

指标体系不是一成不变的，需要随着平台发展持续优化：

1. **提出调整**：任何团队成员发现指标不合理时，在月度评审会提出
2. **影响评估**：评估调整对历史数据对比、看板、告警的影响
3. **讨论决策**：在月度/季度评审会上讨论，达成共识
4. **实施变更**：更新指标定义文档、采集逻辑、看板配置
5. **记录变更**：在指标变更日志中记录变更原因和生效时间

**指标调整原则**：
- 北极星指标一年内不宜更换，除非发现根本性定义缺陷
- 分层指标可按季度微调，但需保持历史可比性
- 新增指标需同时定义计算公式、数据来源和告警阈值

### 6.3 数据质量保障

| 保障措施 | 说明 | 负责人 |
|---------|------|-------|
| **数据完整性检查** | 定时校验采集管道是否正常，检测数据缺失 | 工程团队 |
| **异常值检测** | 对关键指标设置合理范围，超出范围自动标记 | 工程团队 |
| **口径一致性审核** | 确保不同看板对同一指标的计算逻辑一致 | 产品团队 |
| **采集延迟监控** | 监控数据从产生到可查询的延迟，确保时效性 | 工程团队 |
| **变更审计** | 指标定义或采集逻辑的变更需经过 review | 产品 + 工程 |

**数据质量目标**：
- 核心指标数据完整率 >= 99.9%
- 数据采集延迟 <= 5 分钟（实时指标）、<= 1 小时（聚合指标）
- 看板数据与源数据误差 <= 1%

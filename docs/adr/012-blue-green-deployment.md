# ADR-012: 蓝绿部署引入时机评估 — 50 并发规模下的部署策略

- **日期**: 2026-02-14
- **状态**: 已采纳
- **关联**: ADR-003 (基础设施), roadmap v1.4 (蓝绿部署从 Phase 3 推迟到 Phase 4)

## 背景

### 当前部署架构

项目部署在 AWS ECS Fargate 上，使用 ECS 默认滚动更新策略。当前基础设施配置如下：

| 环境 | CPU | 内存 | 任务数 | 部署策略 |
|------|:---:|:----:|:-----:|---------|
| Dev | 256 | 512 MiB | 1 | 滚动更新 (默认) |
| Prod | 512 | 1024 MiB | 2 | 滚动更新 (默认) |

ECS 滚动更新默认参数：`minimumHealthyPercent=100, maximumPercent=200`（ECS 默认值）。ALB 健康检查配置：`interval=30s, timeout=5s, healthyThreshold=2, unhealthyThreshold=3`。

### 决策驱动

roadmap v1.4 将蓝绿部署从 Phase 3 (M9) 推迟到 Phase 4，原因是"初期 ECS 滚动更新足够，用户量小"。现在需要正式评估在 Phase 4 目标用户规模（50+ 活跃用户）下，是否需要引入蓝绿部署，以及引入的最佳时机。

### 关键约束

- **用户规模**: Phase 4 目标 50+ 活跃用户（ADR-007 从 200 人下调）
- **环境策略**: Dev + Prod 两套环境，无 Staging（v1.4 简化）
- **CDK Stack 数量**: 12 个 Stack（6 类 x 2 环境）
- **部署频率**: 预计每周 1-2 次 Prod 部署
- **API 可用性目标**: >= 99.9%（Phase 4 验收标准）

## 备选方案

### 方案 A: 蓝绿部署 (CodeDeploy + ECS)

使用 AWS CodeDeploy 实现 ECS 蓝绿部署，支持流量灰度切换和自动回滚。

**实现要点**：
- 新增 CodeDeploy ECS Deployment Group
- 新增 Green Target Group + Test Listener (端口 8443)
- 部署策略: `LINEAR_10PERCENT_EVERY_1MINUTES`（10 分钟完成切换）
- 自动回滚: `failedDeployment=true, stoppedDeployment=true`

| 优势 | 劣势 |
|------|------|
| 零停机部署，新旧版本并行运行 | 部署期间需要双倍 ECS 任务资源 |
| 支持灰度发布 (Canary/Linear) | 增加 CDK 复杂度 (~200 行代码) |
| 自动回滚，部署失败立即切回 | 需要额外的 Target Group 和 Test Listener |
| 可通过 Test Listener 预验证 | CodeDeploy 与 ECS 滚动更新互斥，切换成本高 |

### 方案 B: 滚动更新增强 (当前方案优化)

在 ECS 默认滚动更新基础上，优化健康检查、部署断路器和 ALB 连接排空配置。

**实现要点**：
- ECS 部署参数调优: `minimumHealthyPercent=50, maximumPercent=200`
- 启用 ECS Deployment Circuit Breaker
- ALB Target Group 连接排空: `deregistrationDelay=30s`
- 健康检查优化: `startPeriod=60s, interval=15s`

| 优势 | 劣势 |
|------|------|
| 零额外资源成本 | 部署期间存在短暂容量下降 (50%) |
| 实现简单 (~30 行代码修改) | 无灰度发布能力 |
| ECS 原生支持，无额外服务依赖 | 回滚需要重新部署上一版本镜像 |
| Deployment Circuit Breaker 提供自动回滚 | 无 Test Listener 预验证能力 |

### 方案 C: 维持现状

保持 ECS 默认滚动更新配置，不做任何调整。

| 优势 | 劣势 |
|------|------|
| 零改动成本 | 默认 `minimumHealthyPercent=100` 导致部署需等待新任务健康后才替换旧任务 |
| 最简单的运维模型 | 无 Deployment Circuit Breaker，部署失败不会自动回滚 |
| - | ALB 默认 `deregistrationDelay=300s`，连接排空时间过长 |
| - | 部署速度较慢（健康检查未优化） |

## 决策

**选择方案 B: 滚动更新增强**。不引入蓝绿部署，而是在现有 ECS 滚动更新基础上优化部署参数。

具体增强项：

1. **ECS 部署参数调优**: `minimumHealthyPercent=50, maximumPercent=200`
2. **启用 Deployment Circuit Breaker**: 部署失败时自动回滚
3. **ALB 连接排空优化**: `deregistrationDelay` 从默认 300s 缩短到 30s
4. **容器健康检查优化**: `startPeriod=60s`（已配置）保持不变，`interval` 从 30s 缩短到 15s

## 方案对比

|------|:-----------:|:-------------:|:-----------:|
| 零停机部署 | 完全零停机 | 接近零停机 (50% 容量保持) | 接近零停机 (100% 容量保持但慢) |
| 灰度发布 | Linear/Canary | 不支持 | 不支持 |

| CDK 改动量 | ~200 行新增 | ~30 行修改 | 0 行 |

### 50 并发场景可用性影响

假设 50 个用户在工作时间内均匀分布，部署期间的影响分析：

| 指标 | 方案 A: 蓝绿 | 方案 B: 滚动增强 | 方案 C: 维持现状 |
|------|:-----------:|:-------------:|:-----------:|
| 部署期间活跃任务数 | 2→4→2 (新旧并行) | 1→3→2 (先缩后扩) | 2→3→2 (先扩后缩) |
| 最小可用容量 | 100% (2 任务) | 50% (1 任务) | 100% (2 任务) |
| 单任务并发处理能力 | ~25 请求/秒 (FastAPI async) | ~25 请求/秒 | ~25 请求/秒 |
| 50 用户时最低可用性 | 无影响 | 1 任务仍可处理 50 并发 | 无影响 |
| 用户可感知的中断 | 无 | 无 (容量充足) | 无 (但部署慢) |
| 连接排空影响 | CodeDeploy 管理 | 30s (优化后) | 300s (默认,过长) |

**关键判断**: 50 并发用户场景下，单个 ECS 任务 (512 CPU/1024 MiB) 运行 FastAPI 异步服务，处理 50 个并发请求完全不成问题。方案 B 的 `minimumHealthyPercent=50%`（即 1 个任务保持运行）在此规模下不会造成任何用户可感知的服务中断。

## 理由

### 1. 用户规模不支撑蓝绿部署的投入

50 活跃用户是一个较小的规模。按工作时间分布，峰值并发请求可能在 10-20 个。单个 ECS 任务 (512 CPU/1024 MiB) 运行 FastAPI 异步框架即可轻松处理这一负载。滚动更新期间保持 1 个任务运行（50% 容量），对 50 用户无任何可感知影响。

### 2. 蓝绿部署的真正价值在高并发场景

蓝绿部署的核心价值是：
- **高并发场景**: 数百到数千用户时，50% 容量下降可能导致请求排队或超时
- **灰度发布**: 需要逐步验证新版本在部分用户上的表现
- **即时回滚**: 生产环境不容忍任何服务中断

对于 50 用户的内部平台，这些价值的边际收益很低：
- 50% 容量下降 = 1 个任务，仍能轻松处理全部请求
- 灰度发布对内部团队的意义不大（可以直接全量发布 + 内部沟通）
- Deployment Circuit Breaker 已提供自动回滚能力

### 3. 滚动增强方案解决了实际痛点

当前默认配置存在的实际问题及解决方案：

| 问题 | 当前默认值 | 增强后 | 效果 |
|------|-----------|--------|------|
| 部署失败无自动回滚 | 无 Circuit Breaker | 启用 Circuit Breaker | 自动检测并回滚失败部署 |
| 连接排空时间过长 | 300s | 30s | 部署速度从 ~8 分钟缩短到 ~3 分钟 |
| 部署速度慢 | minimumHealthy=100% | minimumHealthy=50% | 允许更快地替换旧任务 |

### 4. 保持架构简洁性

当前 12 个 CDK Stack 的复杂度已经较高。引入蓝绿部署需要：
- 新增 Green Target Group
- 新增 Test Listener (端口 8443)
- 配置 CodeDeploy Deployment Group
- 修改 CI/CD Pipeline 适配 CodeDeploy

这些变更增加运维复杂度，但对 50 用户场景几乎没有收益。保持架构简洁，将复杂度预算留给更有价值的能力（如 Phase 4 的执行层分离 P3-4/P3-5）。

## 风险

### 方案 B 已识别风险

| # | 风险 | 概率 | 严重度 | 缓解策略 |
|---|------|:----:|:------:|---------|
| 1 | `minimumHealthyPercent=50%` 期间遇到突发高负载 | 低 | 低 | 50 用户场景下单任务可处理全部请求；部署窗口选在低峰期 |
| 2 | Deployment Circuit Breaker 误判导致回滚 | 低 | 中 | 健康检查 `startPeriod=60s` 给予新任务足够启动时间 |
| 3 | 部署期间数据库迁移导致新旧版本不兼容 | 中 | 高 | Alembic 迁移脚本保持向后兼容；先迁移数据库再部署新版本 |

### 蓝绿部署重新评估触发条件

以下任一条件满足时，应重新评估引入蓝绿部署：

| 触发条件 | 说明 |
|---------|------|
| 活跃用户 > 200 人 | 单任务无法处理峰值并发，50% 容量下降有可感知影响 |
| Prod 任务数 > 4 个 | 滚动更新期间资源波动影响扩大 |
| API P99 延迟 > 1s | 部署期间容量下降可能导致延迟飙升 |
| 部署频率 > 每日 1 次 | 频繁部署增加服务中断风险 |
| 合规要求零停机 | 外部审计要求 SLA >= 99.99% |

## 影响

### 对 CDK 代码的修改

需要修改以下文件：

1. **`infra/lib/constructs/ecs/ecs-service.construct.ts`**: ECS FargateService 添加 `circuitBreaker: { enable: true, rollback: true }`
2. **`infra/lib/constructs/alb/alb.construct.ts`**: Target Group 添加 `deregistrationDelay: cdk.Duration.seconds(30)`
3. **对应的测试文件**: 更新 Fine-grained Assertions

预计改动量: ~30 行代码 + ~20 行测试。

### 对 roadmap.md 的影响

roadmap v1.4 中"蓝绿部署推迟到 Phase 4"的决策进一步明确为"Phase 4 不引入蓝绿部署"。蓝绿部署从 Phase 4 范围中正式移除，待触发条件满足后按需评估。

### 对 deployment.md 的影响

`infra/.claude/rules/deployment.md` §4 蓝绿部署章节保留为参考文档（记录实现方式），但标注为"待未来需要时启用"。新增滚动更新增强配置说明。

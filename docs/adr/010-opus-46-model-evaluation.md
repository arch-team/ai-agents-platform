# ADR-010: Opus 4.6 模型集成评估 — 模型选择策略与成本分析

- **日期**: 2026-02-13
- **状态**: 已采纳
- **关联**: ADR-006 (Agent 框架选型), ADR-008 (Agent Teams 编排策略)

## 背景

### Opus 4.6 可用性

2026-02-05，Claude Opus 4.6 在 Amazon Bedrock 上正式可用（模型 ID: `us.anthropic.claude-opus-4-6-20260205-v1:0`）。Opus 4.6 是 Anthropic 当前最强模型，关键能力包括：

- **1M context window**: 支持超长文档分析和复杂代码库理解
- **Adaptive thinking**: 自适应推理深度，按需分配计算资源
- **复杂推理增强**: 数学、逻辑、多步推理能力显著提升
- **Agentic 任务优化**: 针对工具调用、多轮对话、Agent Loop 场景优化

### 当前模型配置

项目已在 `backend/src/shared/domain/constants.py` 中定义三个模型常量：

```python
MODEL_CLAUDE_HAIKU_35 = "us.anthropic.claude-3-5-haiku-20241022-v1:0"   # 默认
MODEL_CLAUDE_SONNET_4 = "us.anthropic.claude-sonnet-4-20250514-v1:0"
MODEL_CLAUDE_OPUS_46  = "us.anthropic.claude-opus-4-6-20260205-v1:0"

AGENT_DEFAULT_MODEL_ID = MODEL_CLAUDE_HAIKU_35  # 默认使用 Haiku
```

用户通过 `AgentConfig.model_id` 字段选择模型，切换只需修改该配置值，无需代码变更。

### 决策驱动

需要评估是否应将默认模型升级为 Opus 4.6，或制定更精细的模型选择策略，以在能力和成本之间取得平衡。

## 备选方案

### 策略 A: 维持现状 — 默认 Haiku + 手动切换

保持 `AGENT_DEFAULT_MODEL_ID = MODEL_CLAUDE_HAIKU_35`，用户通过 Agent 配置界面自行选择模型。

| 优势 | 劣势 |
|------|------|
| 成本最低，默认场景 Token 费用可控 | 用户可能不了解何时该切换模型 |
| 无需代码变更 | 复杂任务使用 Haiku 可能效果不佳 |
| 用户拥有完全控制权 | 缺乏引导，模型选择依赖用户经验 |

### 策略 B: 智能路由 — 根据任务复杂度自动选择模型

新增模型路由层，根据输入长度、工具数量、任务类型等特征自动选择模型。

| 优势 | 劣势 |
|------|------|
| 自动优化性价比 | 需要新增路由逻辑（~200-300 行代码） |
| 用户无需了解模型差异 | 路由规则难以覆盖所有场景 |
| 复杂任务自动升级模型 | 引入额外延迟（路由判断） |
| - | 路由错误可能导致成本超预期或质量不足 |

### 策略 C: 全面升级 — 默认改为 Opus 4.6

将 `AGENT_DEFAULT_MODEL_ID` 改为 `MODEL_CLAUDE_OPUS_46`，所有 Agent 默认使用最强模型。

| 优势 | 劣势 |
|------|------|
| 所有场景获得最佳质量 | Token 成本增加 15 倍 |
| 用户无需关心模型选择 | 简单任务严重浪费资源 |
| Agent Teams 获得最强推理能力 | 50+ 用户规模下月度成本不可控 |
| - | 响应延迟增加（Opus 推理更慢） |

## 决策

**选择策略 A: 维持现状**（默认 Haiku 3.5），并补充模型选择指南文档，帮助用户按场景选择合适的模型。

## 理由

### 1. 成本差异决定了默认策略

三个模型的定价对比（美元/百万 Token，近似值，实际以 AWS 控制台为准）：

| 模型 | 输入价格 | 输出价格 | 成本倍数 (vs Haiku) |
|------|:--------:|:--------:|:-------------------:|
| Haiku 3.5 | $1.00 | $5.00 | **1x** |
| Sonnet 4 | $3.00 | $15.00 | **3x** |
| Opus 4.6 | $15.00 | $75.00 | **15x** |

以典型 Agent 对话为例（输入 2K tokens + 输出 1K tokens）：

| 模型 | 单次对话成本 | 月均成本 (50 用户, 20 次/天) |
|------|:----------:|:-------------------------:|
| Haiku 3.5 | ~$0.007 | ~$210 |
| Sonnet 4 | ~$0.021 | ~$630 |
| Opus 4.6 | ~$0.105 | ~$3,150 |

对于企业内部平台（50+ 用户规模），默认 Opus 意味着月度 Token 成本从 ~$210 增长到 ~$3,150，增幅 15 倍。大多数使用场景（简单问答、信息提取、分类）并不需要 Opus 级别的推理能力。

### 2. Agent Teams 场景成本敏感

ADR-008 记录了 Agent Teams 的 Token 消耗特征：

- 3 人团队单次执行约消耗 ~800K tokens
- `max_turns=200`，多轮交互进一步放大消耗
- `asyncio.Semaphore(3)` 限制并发，但不限制单次消耗

Agent Teams 场景下的模型成本对比：

| 模型 | 单次团队执行 (~800K tokens) | 月均成本 (10 次/天) |
|------|:-------------------------:|:-------------------:|
| Haiku 3.5 | ~$2.40 | ~$720 |
| Sonnet 4 | ~$7.20 | ~$2,160 |
| Opus 4.6 | ~$36.00 | ~$10,800 |

Agent Teams 使用 Opus 4.6 的月成本可达 $10,800，不适合作为默认配置。建议 Agent Teams 场景优先考虑 Sonnet 4（性价比均衡）。

### 3. 灵活性已具备

当前架构已支持用户按需切换模型：

- 三个模型常量已在 `constants.py` 中定义
- `AgentConfig.model_id` 字段允许每个 Agent 独立配置模型
- 前端配置向导已包含模型选择（M11 #11）
- 切换模型无需修改代码，只需更改配置值

无需引入智能路由的额外复杂度，用户可根据场景自行决策。

### 4. 策略 B 的投入产出比不高

智能路由（策略 B）需要：
- 新增路由判断逻辑和评估规则
- 维护路由规则的准确性
- 处理路由错误的边界情况

对当前项目阶段而言，通过模型选择指南引导用户手动选择，比自动路由更可控、更透明。未来如用户规模和使用复杂度增长，可重新评估智能路由方案。

## 模型选择指南

### 按使用场景推荐

| 场景 | 推荐模型 | 理由 |
|------|---------|------|
| 简单问答 / FAQ | Haiku 3.5 | 低成本、低延迟，简单任务性能足够 |
| 信息提取 / 分类 | Haiku 3.5 | 结构化提取任务不需要深度推理 |
| 代码生成 / 文档撰写 | Sonnet 4 | 均衡的性价比，中等复杂度任务表现良好 |
| Agent Teams 协作 | Sonnet 4 | max_turns=200 导致 Token 消耗大，Opus 成本过高 |
| 通用对话 / 客户服务 | Sonnet 4 | 流畅自然的对话能力，成本可接受 |
| 复杂推理 / 数学证明 | Opus 4.6 | 最强推理能力，值得额外成本投入 |
| 长文档分析 (>100K tokens) | Opus 4.6 | 1M context window，其他模型无法有效处理超长上下文 |
| 安全审计 / 合规检查 | Opus 4.6 | 需要最高准确率，错误成本远超 Token 成本 |
| 复杂代码审查 / 架构分析 | Opus 4.6 | 需要深度理解大规模代码库和架构关系 |

### 决策流程图

```
收到任务
  ├─ 简单任务 (问答/分类/提取)?
  │   └─ ✅ Haiku 3.5
  ├─ 中等任务 (代码生成/对话/文档)?
  │   └─ ✅ Sonnet 4
  ├─ Agent Teams 协作?
  │   └─ ✅ Sonnet 4 (成本敏感)
  ├─ 复杂推理 / 长文档 / 安全审计?
  │   └─ ✅ Opus 4.6
  └─ 不确定?
      └─ 从 Sonnet 4 开始, 按需升级
```

### 成本控制建议

1. **默认从 Haiku 开始**: 新建 Agent 时默认 Haiku 3.5，验证效果后再考虑升级
2. **Agent Teams 避免 Opus**: 除非任务确实需要最强推理，否则 Teams 场景使用 Sonnet 4
3. **监控 Token 用量**: 关注高消耗 Agent，评估是否模型选择合理
4. **按场景模板推荐**: 后续可在 Agent 创建流程中根据场景模板预设推荐模型

## 影响

### 1. 无需代码变更

本决策不涉及代码修改。三个模型常量和 `AgentConfig.model_id` 配置机制已就位，维持现状即可。

### 2. 前端配置已支持

M11 #11 已在 Agent 配置向导中新增模型成本对比说明，用户可在创建 Agent 时查看各模型的能力与成本对比。

### 3. 文档交付

本 ADR 的模型选择指南章节即为交付物，供用户和开发团队参考。

### 4. 后续演进

| 演进方向 | 触发条件 | 预期方案 |
|---------|---------|---------|
| 场景模板推荐 | Agent 创建时需要模型引导 | 在 Agent 模板中预设推荐模型 |
| 智能路由 (策略 B) | 用户规模 >200 或误选模型率 >30% | 引入轻量路由层自动推荐 |
| Token 预算控制 | 月度成本超预期 | 在 Agent/Team 级别增加 Token 预算上限 |
| 模型评测基准 | 需要量化模型效果差异 | 建立内部评测集，对比三个模型在关键场景的表现 |

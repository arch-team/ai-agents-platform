## 2. 经验教训

### 架构层面

| 经验 | 说明 | 后续建议 |
|------|------|---------|
| **SDK 能力 > 自建引擎** | Claude Agent SDK 的 Teams 特性已内置团队创建/任务分配/消息协调。自建 DAG 引擎是在重复造轮子 | 优先探索 SDK/平台原生能力，仅在缺失时自建 |
| **模块内扩展 > 新建模块** | Agent Teams 在 execution 模块内扩展 (~1500 行)，比新建 orchestration 模块 (~3000 行) 更轻量且耦合更低 | 功能增强优先考虑扩展现有模块 |
| **异步任务 session 隔离** | 后台 asyncio.Task 使用请求级 DB session 导致操作失败，需要独立 session | 所有超出请求生命周期的后台任务必须使用独立 session |
| **execute_stream 调用约定** | `async def` 返回 AsyncIterator 需先 `await` 再 `async for` | SDK 接口的异步约定需要单元测试覆盖 |

### 开发效率

| 经验 | 说明 |
|------|------|
| **并行 Agent 开发** | 3 个 Agent 同时实现领域层/服务层/测试层，在有明确接口约定时效率很高 |
| **TDD 保障重构信心** | 1595 测试使得修复 session 隔离、stream 调用等 Bug 时不担心回归 |
| **端到端验证前移** | 端到端验证发现了 2 个仅靠单元测试无法发现的 Bug (session 生命周期 + stream 调用) |

## 4. 用户反馈

**当前状态: 无实际终端用户**。后端 66+ 端点已就绪，但前端完全缺失。这是项目产品化的最大瓶颈。

| 反馈维度 | 现状 | 影响 |
|---------|------|------|
| 前端 UI | 零实现 | 无法验证用户体验，无法收集真实反馈 |
| 核心团队试用 | 仅通过 curl/API 测试 | 验收标准中的"20+ 活跃用户"无法达成 |
| Agent Templates | 10 个预置模板已定义 | 无 UI 展示，模板价值无法体现 |

## 5. 路线图调整决策

### 5.1 orchestration 模块正式取消

| 决策 | 理由 |
|------|------|
| 取消独立 `orchestration` 模块 | ADR-008: Agent Teams 已覆盖 Multi-Agent 协作场景；DAG 引擎的"预定义工作流"场景当前无明确业务需求 |
| 移除前端 DAG 编辑器需求 | 依附于 orchestration 模块，随之取消 |
| 如未来需要固定工作流 | 可在 TeamExecution 基础上叠加 WorkflowTemplate 概念，无需独立模块 |

### 5.2 前端开发提升为最高优先级

**核心论点**: 后端 8 个模块 66+ 端点已就绪，但零前端意味着零用户。Phase 3 验收标准"20+ 活跃用户"的前提是有可用的 UI。

**调整**: 将前端 MVP 开发插入 M8 和 M9 之间，作为 M8.5 或与 M8 并行。

### 5.3 M8 evaluation 模块简化

Agent Teams 的输出质量主要依赖 Claude 模型能力，短期内自建评估框架的 ROI 有限。建议:
- 缩减 M8 范围: 仅实现"测试集定义 + 批量执行"，不含复杂评分体系
- 将节省的容量投入前端开发

### 5.5 Phase 3 验收标准调整

| 指标 | 原目标 | 调整后 | 理由 |
|------|--------|--------|------|
| 编排工作流支持 | 串行/并行/条件分支 | **Agent Teams 自主编排** | ADR-008: 不再要求显式 DAG 模式 |
| 核心团队活跃用户 | >= 20 人 | **>= 10 人** | 前端新上线初期，降低预期 |
| 前端功能覆盖 | 可视化编排器 | **登录 + Agent 管理 + 对话 + 团队执行** | 替换 DAG 编辑器为实际可用的核心流程 |
| Agent 评估覆盖率 | 所有公开模板 | **>= 3 个核心模板有基线评估** | 评估模块精简后降低覆盖目标 |
| API 可用性 | >= 99.5% (Prod) | 不变 | - |
| P95 响应延迟 | < 300ms (非 LLM) | 不变 | - |
| AgentCore P2 | 3/3 | ✅ 已完成 | - |

### 5.6 Phase 4 影响

Phase 3 后剩余 Phase 4 工作:
- `audit` 模块: 不变
- `insights` 增强: 不变
- AgentCore P3 (3 项): 推迟到 Phase 4
- 前端迭代: 自助式 Agent 构建器移到 Phase 4

## 结论

M7 通过 Agent Teams 大幅简化了 Multi-Agent 编排实现，节省了预计 6-8 周的 DAG 引擎开发工作量。这些容量应投入到前端 MVP 开发 (当前最大的产品缺口) 和生产环境部署。Phase 3 剩余里程碑从 2 个(M8+M9) 重新编排为 3 个 (M8 精简 + M8.5 前端 + M9 部署)，总时间窗口从原 24 周缩短到 16 周。

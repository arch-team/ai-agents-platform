## 1. 反馈收集方案

### 1.1 反馈表单字段设计

以下为反馈表单的标准字段结构，适用于飞书表单和 Google Form 两种方案。

| 字段名 | 类型 | 必填 | 选项/说明 |
|--------|------|:----:|---------|
| 问题类型 | 单选下拉 | 是 | Bug 报告 / 功能建议 / 体验问题 / 使用疑问 / 其他 |
| 严重程度 | 单选下拉 | 是 | P0 紧急（平台不可用）/ P1 高（核心功能异常）/ P2 中（体验问题）/ P3 低（建议或非阻塞） |
| 问题标题 | 单行文本 | 是 | 一句话描述问题（限 100 字） |
| 详细描述 | 多行文本 | 是 | 包括操作步骤、预期结果、实际结果 |
| 使用场景 | 单选下拉 | 否 | 创建 Agent / 对话交互 / Agent 管理 / 知识库上传 / 权限设置 / 仪表板查看 / 其他 |
| 截图/录屏 | 文件上传 | 否 | 支持 PNG/JPG/GIF/MP4，单文件不超过 20MB |
| 浏览器信息 | 单行文本 | 否 | 如 Chrome 120、Safari 17 |
| 联系方式 | 单行文本 | 是 | 飞书/企业微信/Slack 账号，用于跟进沟通 |
| 姓名/部门 | 单行文本 | 是 | 便于按部门统计反馈分布 |

### 1.2 飞书表单设置步骤

**前置条件**: 拥有飞书管理后台权限或多维表格编辑权限。

**方案 A: 飞书多维表格 + 表单视图（推荐）**

1. **创建多维表格**
   - 打开飞书 → 文档 → 新建多维表格
   - 命名为「AI Agents Platform - 用户反馈收集」

2. **配置字段**
   - 按照 §1.1 字段设计表，逐一添加列:
     - "问题类型"：选择"单选"字段类型，添加 5 个选项
     - "严重程度"：选择"单选"字段类型，添加 P0/P1/P2/P3 四个选项
     - "问题标题"：选择"文本"字段类型
     - "详细描述"：选择"文本"字段类型，开启"多行文本"
     - "使用场景"：选择"单选"字段类型，添加 7 个选项
     - "截图/录屏"：选择"附件"字段类型
     - "浏览器信息"：选择"文本"字段类型
     - "联系方式"：选择"文本"字段类型
     - "姓名/部门"：选择"文本"字段类型
   - 额外添加管理字段:
     - "处理状态"：单选（待处理 / 处理中 / 已修复 / 已关闭 / 延期处理）
     - "处理人"：成员字段
     - "提交时间"：日期字段，开启"自动填入创建时间"
     - "修复时间"：日期字段

3. **创建表单视图**
   - 点击视图栏右侧"+"→ 选择"表单视图"
   - 设置表单标题: 「AI Agents Platform 问题反馈」
   - 设置表单描述（见下方模板）
   - 设置必填字段: 问题类型、严重程度、问题标题、详细描述、联系方式、姓名/部门
   - 隐藏管理字段（处理状态、处理人、修复时间），不在表单中展示

4. **发布表单**
   - 点击"分享"→ 生成表单链接
   - 设置权限为"组织内所有人可填写"
   - 将链接发送至支持群

**表单描述模板**:

```
感谢您参与 AI Agents Platform Wave 1 种子用户测试！

请通过此表单提交您在使用过程中遇到的问题或建议。
我们承诺在 24 小时内响应您的反馈。

严重程度参考：
- P0 紧急：平台完全不可用、数据丢失
- P1 高：核心功能异常（如对话失败、Agent 创建失败）
- P2 中：体验问题（如界面不友好、加载缓慢）
- P3 低：功能建议、优化建议
```

**方案 B: 飞书问卷（简易方案）**

1. 飞书 → 文档 → 新建问卷
2. 逐一添加 §1.1 中的字段
3. 发布后获取链接

> 注意: 飞书问卷的附件上传和数据管理能力弱于多维表格方案，建议优先使用方案 A。

### 1.3 Google Form 设置步骤（备选方案）

适用于团队主要使用 Google Workspace 的场景。

1. **创建表单**
   - 访问 Google Forms
   - 新建空白表单，标题设为「AI Agents Platform 问题反馈」

2. **添加字段**
   - 问题类型: 选择"下拉菜单"，添加 5 个选项
   - 严重程度: 选择"下拉菜单"，添加 P0-P3 四个选项
   - 问题标题: 选择"简答题"，添加字数限制
   - 详细描述: 选择"段落题"
   - 使用场景: 选择"下拉菜单"
   - 截图/录屏: 选择"文件上传"，限制文件类型和大小
   - 浏览器信息: 选择"简答题"
   - 联系方式: 选择"简答题"
   - 姓名/部门: 选择"简答题"

3. **表单设置**
   - 设置 → 回复 → 开启"收集电子邮件地址"
   - 设置 → 回复 → 开启"向回复者发送回复副本"
   - 将回复关联到 Google Sheet 以便数据分析

4. **发布**
   - 点击"发送"→ 复制链接
   - 分享至支持群

### 1.4 反馈分级标准

对齐 `rollout-plan.md` 中的问题分级定义:

| 级别 | 定义 | 典型场景 | 响应时间 | 修复时间 |
|------|------|---------|---------|---------|
| **P0 紧急** | 平台不可用、数据丢失 | 服务宕机、无法登录、对话数据丢失、安全漏洞 | **1 小时** | **4 小时** |
| **P1 高** | 核心功能异常 | 对话发送失败、Agent 创建失败、知识库上传异常、权限错误 | **4 小时** | **1 个工作日** |
| **P2 中** | 体验问题 | UI 显示异常、加载缓慢（> 5 秒）、操作不符合直觉、缺少提示信息 | **1 个工作日** | **1 周** |
| **P3 低** | 功能建议 | 新功能请求、非阻塞优化建议、文档补充需求 | **1 周** | **下个迭代** |

**分级判定原则**:

- 影响所有用户 → 至少 P1
- 影响数据安全 → 至少 P0
- 有替代方案可绕过 → 降一级
- 多人反馈同一问题 → 升一级

### 1.5 反馈处理 SOP 流程

```
收集 → 分级 → 分配 → 修复 → 回复 → 闭环
```

**详细流程**:

#### 步骤 1: 收集

| 操作 | 负责人 | 时间要求 |
|------|--------|---------|
| 检查反馈表单新提交 | 值班人员 | 每日上午 10:00 |
| 检查支持群中的问题消息 | 值班人员 | 收到后 4 小时内 |
| 将群内问题引导至表单提交 | 值班人员 | 即时 |

#### 步骤 2: 分级

| 操作 | 负责人 | 说明 |
|------|--------|------|
| 确认问题可复现 | 值班人员 | 尝试复现用户描述的问题 |
| 按 §1.4 标准确定级别 | 值班人员 | 不确定时偏向高级别 |
| 填写处理状态为"处理中" | 值班人员 | 在多维表格/Sheet 中更新 |

#### 步骤 3: 分配

| 级别 | 分配策略 |
|------|---------|
| P0 | 立即通知技术负责人 + 在群内通报 |
| P1 | 分配给对应模块负责人，当日处理 |
| P2 | 记录到迭代计划，按排期处理 |
| P3 | 汇总到需求池，周会讨论优先级 |

#### 步骤 4: 修复

| 操作 | 说明 |
|------|------|
| 创建修复分支 | 命名规范: `fix/feedback-{编号}-{简述}` |
| 编写修复代码 + 测试 | 遵循项目 TDD 规范 |
| 代码审查 | P0/P1 需至少 1 人 Review |
| 部署验证 | 在测试环境验证后部署到生产 |

#### 步骤 5: 回复

| 操作 | 方式 | 说明 |
|------|------|------|
| 通知提交者问题已修复 | 支持群 @用户 或飞书私信 | 使用回复模板 |
| 提供修复说明 | 简述修复内容和影响 | — |
| 确认用户验证通过 | 请用户确认问题已解决 | — |

**回复模板**:

```
Hi {用户名}，

您提交的问题「{问题标题}」已修复。

修复内容: {简述修复了什么}
当前状态: 已部署到生产环境

请您验证问题是否已解决。如仍有问题，请回复告知，我们会继续跟进。

感谢您的反馈！
```

#### 步骤 6: 闭环

| 操作 | 说明 |
|------|------|
| 用户确认已解决 | 更新表单状态为"已修复" |
| 超过 3 天无回复 | 自动标记为"已关闭" |
| 记录修复耗时 | 用于统计平均修复时间 |
| 纳入周报统计 | 见 §1.6 周度汇总模板 |

### 2.3 群管理规范

#### 响应时间承诺

| 问题类型 | 响应时间 | 处理方式 |
|----------|---------|---------|
| P0 紧急（平台不可用） | 1 小时 | 立即处理，群内通报进展 |
| P1 高（功能异常） | 4 小时 | 确认问题后群内回复，当日修复 |
| P2 中（体验问题） | 1 个工作日 | 群内回复已收到，排入计划 |
| P3 低（建议） | 1 周 | 汇总后统一回复 |
| 使用疑问 | 4 小时 | 群内直接解答或提供文档链接 |

#### 值班安排建议

| 项目 | 安排 |
|------|------|
| 值班周期 | 每周轮换 |
| 值班人数 | 每周 1 名主值班 + 1 名备份 |
| 值班职责 | 每日检查反馈表单、响应群消息、更新反馈状态 |
| 交接方式 | 每周一上午交接，传递未关闭问题清单 |
| 值班记录 | 在反馈表格中记录值班日志 |

**值班日志模板**:

```
【值班日志】{日期}
值班人: {姓名}
新增反馈: {N} 条
已处理: {N} 条
待跟进: {N} 条（编号: F-xxx, F-xxx）
备注: {特殊情况说明}
```

#### 群内沟通规范

- 技术团队成员尽量在 4 小时内响应用户提问
- 对于无法立即解决的问题，先回复"已收到，正在排查"
- 引导用户将详细 Bug 描述提交到反馈表单
- 避免在群内讨论敏感技术细节（如安全漏洞），转至内部频道处理
- 修复完成后在群内通报，@相关用户确认

## 3. 运营节奏

### 3.1 每日运营

| 时间 | 事项 | 负责人 | 说明 |
|------|------|--------|------|
| 上午 10:00 | 检查反馈表单新提交 | 值班人员 | 浏览新增反馈，完成初步分级 |
| 上午 10:00 | 回复群内过夜消息 | 值班人员 | 对隔夜未回复的消息进行响应 |
| 全天 | 群消息响应 | 值班人员 | 工作时间内 4 小时响应 |
| 全天 | P0/P1 问题跟进 | 值班人员 + 技术负责人 | 按 SOP 流程处理 |
| 下班前 | 更新当日值班日志 | 值班人员 | 记录当日反馈处理情况 |

### 3.2 每周运营

| 时间 | 事项 | 参与人 | 产出 |
|------|------|--------|------|
| 周一上午 | 值班交接 | 上周/本周值班人员 | 未关闭问题清单交接 |
| 周三/周四 | 反馈汇总会 | 技术团队 + 运营 | 周度反馈汇总报告（§1.6 模板） |
| 周五下午 | 问题修复通报 | 值班人员 | 在群内通报本周修复清单 |
| 周五下午 | 下周计划沟通 | 技术团队 | 确定下周需修复的 P2 问题 |

### 3.3 Wave 1 结束时运营

Wave 1 结束标准: 10 人全部完成 Onboarding + 各自创建至少 1 个 Agent。

#### 满意度调查

在 Wave 1 结束时（预计 Week 4 结束），向全部 10 名种子用户发送满意度调查。

**调查内容**:

| 题目 | 类型 | 选项 |
|------|------|------|
| 整体满意度 | 评分 | 1-5 分（1=非常不满意，5=非常满意） |
| 平台易用性 | 评分 | 1-5 分 |
| Agent 创建体验 | 评分 | 1-5 分 |
| 对话交互体验 | 评分 | 1-5 分 |
| 文档和指引质量 | 评分 | 1-5 分 |
| 技术支持响应 | 评分 | 1-5 分 |
| 最满意的功能 | 多选 | 对话交互 / Agent 创建 / 知识库 / 预置模板 / 其他 |
| 最需要改进的方面 | 多选 | 性能速度 / 界面设计 / 功能丰富度 / 文档质量 / 稳定性 / 其他 |
| 是否愿意推荐给同事 | 评分 | 0-10 分（NPS 净推荐值） |
| 其他建议 | 开放文本 | — |
